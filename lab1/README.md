## Вариант 1  – Социальная сеть

Приложение должно содержать данные:
- Пользователь
- Стена
- Сообщения чата (PtP)

Реализовать API:
- Создание нового пользователя
- Поиск пользователя по логину
- Поиск пользователя по маске имя и фамилии
- Добавление записи на стену
- Загрузка стены пользователя
- Отправка сообщения пользователю
- Получение списка сообщений для пользователя


## Описание системы

Социальная сеть предоставляет пользователям возможность создавать профиль, находить других пользователей, публиковать записи на своей стене и обмениваться личными сообщениями.

Пользователи взаимодействуют с системой через веб-клиент или мобильное приложение.  
Запросы поступают через API Gateway и далее обрабатываются специализированными сервисами системы.

Основные подсистемы:

- сервис аутентификации
- сервис управления пользователями
- сервис стены
- сервис чата
- сервис модерации

Для хранения данных используются базы данных PostgreSQL и MongoDB, а также Redis для кэширования.

Система также интегрируется с внешним сервисом отправки уведомлений.


## Основные API системы

Реализуются следующие операции:

- создание нового пользователя
- поиск пользователя по логину
- поиск пользователя по маске имени и фамилии
- добавление записи на стену
- загрузка стены пользователя
- отправка сообщения пользователю
- получение списка сообщений пользователя


## Роли пользователей

В системе предусмотрены следующие роли:

**Гость (Guest)**  
Незарегистрированный пользователь. Ему видны только окна регистрации / входа.

**Пользователь (User)**  
Зарегистрированный пользователь. Может:
- создавать записи на стене
- просматривать свою и чужие стены
- отправлять личные сообщения
- получать сообщения

**Модератор (Moderator)**  
Пользователь с административными правами. Может просматривать жалобы и удалять неподходящий контент.

## Внешние системы

**Сервис уведомлений**

Внешняя система отправки уведомлений пользователям:

- push уведомления
- email уведомления
- SMS уведомления

Используется для уведомления пользователя о новых сообщениях.


## Архитектура системы

Архитектура системы описана в стиле **C4 Model** с использованием **Structurizr DSL**.

Описание архитектуры находится в `workspace.dsl`


## Диаграммы

### C1 — System Context

Диаграмма контекста системы показывает:

- роли пользователей
- взаимодействие пользователей с системой
- интеграцию системы с внешним сервисом уведомлений

### C2 — Container Diagram

Диаграмма контейнеров показывает основные компоненты системы:

- Web Client
- Mobile Client
- API Gateway
- Auth Service
- User Service
- Wall Service
- Chat Service
- Moderation Service
- базы данных
- Redis

Также на диаграмме отображены технологии контейнеров и протоколы взаимодействия.


### Dynamic Diagram

Динамическая диаграмма показывает архитектурно значимый сценарий:

**Отправка личного сообщения пользователю**

Сценарий включает:

1. пользователь отправляет сообщение
2. запрос проходит через API Gateway
3. Chat Service обрабатывает сообщение
4. сообщение сохраняется в базе данных
5. обновляется кэш
6. пользователю отправляется уведомление

Также реализован дополнительный сценарий модерации контента.


## Используемые технологии

Основные технологии системы:

- Web Client — Web UI
- Mobile Client — Mobile UI
- API Gateway — Nginx / Reverse Proxy
- Backend сервисы — FastAPI
- Базы данных — PostgreSQL, MongoDB
- Кэширование — Redis
- Взаимодействие сервисов — HTTPS / REST

## Генерация PlantUML из Structurizr DSL (Docker)

Из корня проекта (где лежит `workspace.dsl`):

``` shell
docker run --rm -v "%cd%":/work structurizr/cli export -workspace /work/workspace.dsl -format plantuml -output /work/puml
docker run --rm -v "%cd%":/work plantuml/plantuml -tpng /work/puml
```
