## Вариант 1  Социальная сеть

Приложение должно содержать данные:
- Пользователь
- Стена
- Сообщения чата (PtP)

Реализовать API:
- Создание нового пользователя
- Поиск пользователя по логину
- Поиск пользователя по маске имя и фамилии
- Добавление записи на стену
- Загрузка стены пользователя
- Отправка сообщения пользователю
- Получение списка сообщений для пользователя

## Архитектура (C4 / Structurizr)

В репозитории описана архитектура в стиле  Architecture as Code  с использованием  Structurizr DSL .

### Роли
-  Гость  – незарегистрированный пользователь: просмотр публичной информации, поиск пользователей.
-  Пользователь  – зарегистрированный: публикации на стене, PtP сообщения.
-  Модератор  – проверка/удаление неподходящего контента.

### Внешние системы
-  Сервис уведомлений (External)  – отправка уведомлений (email/push/SMS) при новом PtP сообщении.

### Контейнеры системы
-  Веб-клиент  – интерфейс в браузере.
-  Мобильный клиент  – интерфейс на телефоне.
-  API Gateway (Nginx)  – единая точка входа, маршрутизация запросов по сервисам.
-  User Service (FastAPI)  – создание и поиск пользователей.
-  Wall Service (FastAPI)  – добавление записи на стену и загрузка стены пользователя.
-  Chat Service (FastAPI)  – отправка PtP сообщений и получение списка сообщений пользователя.
-  PostgreSQL  – хранение пользователей и записей стены.
-  MongoDB  – хранение PtP сообщений.
-  Redis  – кеширование часто запрашиваемых данных (поиск/стена/последние сообщения).

## Диаграммы
В `workspace.dsl` описаны:
- C1 – System Context: пользователи, система, внешние системы.
- C2 – Container: контейнеры внутри системы и их взаимодействие.
- D1 – Dynamic: сценарий отправки PtP сообщения.

## Генерация PlantUML из Structurizr DSL (Docker)

Из корня проекта (где лежит `workspace.dsl`):

``` shell
docker run --rm -v "%cd%":/work structurizr/cli export -workspace /work/workspace.dsl -format plantuml -output /work/puml
```
